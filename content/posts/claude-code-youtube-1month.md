> このシステムは Claude Code (Anthropic) に作ってもらったものです。コードは私が書いていません。この記事も Claude に書いてもらいました。「AI に任せて作ったら、どこで詰まるか」の現場記録として読んでください。

## Claude Code に副業を作らせてみた話

Claude Code に「副業になりそうな仕組みを作って」とは言っていません。実際に頼んだのは「YouTube Shorts を完全自動で投稿するパイプライン」を作ってほしい、というシステムの仕様だけです。

その結果として、宇宙系の解説 Shorts を 1日2回 自動投稿するチャンネルが約1ヶ月動きました。投稿数は約60〜70本、動画あたりの再生数は平均1000回前後（最高で5桁手前、少ないものは1桁台）。爆発的にバズったわけでもなく、収益化条件にも届かず、視聴者からは品質に関する辛いフィードバックを受けました。

それでも記事として残す価値があると思ったのは、**「Claude Code に任せたらどこまでできて、どこができなかったか」** が比較的はっきり見えたからです。

## Claude Code が作ったシステム

### 全体構成

| コンポーネント | 役割 |
|---|---|
| n8n（Docker） | ワークフロー制御 |
| Claude API | 台本（ナレーション原稿）生成 |
| VOICEBOX | 音声合成 |
| FFmpeg | 字幕付き縦動画レンダリング |
| YouTube Data API | 自動投稿 |

実装は全部 Claude です。VOICEBOX と FFmpeg は外部ソフトなので、Claude が書いたのは接続コード（Python HTTP ブリッジ）と字幕合成スクリプトの部分。「Docker のコンテナの中の n8n から、Windows ネイティブの VOICEBOX と FFmpeg を呼ぶブリッジサーバー」みたいな、地味に厄介な部分も含めて全部書いてもらいました。

### Claude が組んだ設計判断（後から振り返って正しかったもの）

- **題材プール方式**: 事前に題材を100件程度ストックしておき、ランダムに1件選んで使う。投稿成功したら使用済みに移す
- **4層の重複防止ガード**: 題材プール段階 / LLM 判定 / 類似度チェック（bigram Jaccard） / 読み辞書補正
- **失敗時のロールバック**: 投稿失敗ならプールに戻す、誤読発見ならその語を辞書に追加
- **Discord 通知での可視化**: 投稿の成否、生成された動画のメタ情報、視聴者コメントの分析結果まで、すべて Discord に Webhook で通知。スマホからどこでも稼働状況とフィードバックが確認できる

これらは私が指示したわけではなく、Claude が「こうした方が良いと思います」と提案してきた構造です。動いてみると、確かに重複事故ゼロで運用できましたし、Discord で動向が見えるので「動いている安心感」もありました。

## 再現するなら、こんな感じで Claude Code に頼めば作れる

実際に使ったプロンプトと完全に同じではありませんが、同じシステムを Claude Code に依頼するならこの程度の指示で組めます。

**全体の依頼:**

```
YouTube Shorts を完全自動投稿するパイプラインを n8n で作ってください。

要件:
- 1日2回の自動実行
- ジャンルは [自分が選んだジャンル] の解説 Shorts
- 台本生成は Claude API（Sonnet）
- 音声合成は VOICEBOX（Docker から host.docker.internal 経由）
- 動画レンダリングは FFmpeg（Windows ネイティブ、Python HTTP ブリッジ経由）
- YouTube Data API で投稿
- 同じ題材の重複を防ぐガード（題材プール / LLM主旨判定 / 類似度チェック）
- 投稿成功/失敗を Discord Webhook に通知

構成図と必要なファイル一覧を最初に提示してから実装に入ってください。
```

**重複防止ガードの追加依頼:**

```
台本生成時に同じ題材が連続しないよう、4層のガードを実装してください:
1. 題材プール段階（normalize で表記ゆれ吸収）
2. LLM で同主旨判定
3. bigram Jaccard 類似度で fail-safe（threshold 0.4）
4. VOICEBOX の読み辞書補正
どこか1層で重複と判定されれば throw。
```

**フィードバック取り込み:**

```
YouTube のコメントを毎日取得して、Claude で:
- センチメント分析
- カテゴリ別の傾向集計
- VOICEBOX の誤読指摘を抽出 → 辞書に自動登録

結果は feedback.json に保存し、Discord に分析レポートを通知してください。
```

このくらいの粒度で投げれば、Claude Code が必要なファイル群を順に作って、各依存関係を揃えてくれます。途中で「ここはどうしますか？」と質問が来るので、その都度方針を答えるだけです。

## Claude Code だけでは完結しない作業

コードは Claude が全部書いてくれますが、以下は自分の手を動かす必要があります。

- Google Cloud Console での YouTube OAuth クライアント作成
- Anthropic / YouTube Data API のキー発行
- VOICEBOX のインストールと起動
- Docker Desktop のセットアップ
- Windows タスクスケジューラへの登録（管理者権限の作業）

ただし「自分でやる」と言っても、コードを書くわけではありません。それぞれ**画面のスクショを Claude に渡しながら「次どうすればいい？」と聞けば、Claude が画面のどこをクリックするかまで含めて手順を教えてくれます**。

実際、Cloud Console で詰まったときは「OAuth 同意画面の設定でこういうエラーが出ている」とスクショを見せて指示を仰ぎました。Claude は画面の文字情報を読み取って、「ここの『公開ステータス』はテストのままで、テストユーザーに自分の Google アカウントを追加してください」というレベルで具体的に返してくれます。

つまり**コーディングの知識がなくても、Claude との対話のしかた（要件の伝え方 / 詰まったときに何をどう見せるか）さえ覚えれば、このパイプラインは誰でも組み立てられる**、という話です。私自身、コードは1行も書いていません。

## 動かしてみて見えた、AIだけでは届かない部分

仕組みは動きました。次の問題は、**動いた結果が良いものかどうか** です。

運用約1ヶ月分の動画コメントをセンチメント分析にかけた結果がこちらです。

> センチメント分析: コメントが肯定的か否定的かを `-1〜+1` の値で示したもの。`-1` が最もネガティブ、`+1` が最もポジティブ、`0` がニュートラル。今回は Claude API にコメントを投げて値を返してもらっています。

### 視聴者からの実フィードバック

**ネガティブ寄り**

カテゴリ:「動画クオリティへの全般的な不満」（センチメント -0.9）
- 「漢字の読み間違いが多く内容に集中できない」
- 「全て過去形で語られており現状が不明瞭」
- 「日本語品質・制作クオリティへの不信感」

**ポジティブ寄り**

カテゴリ:「題材そのものへの関心」
- 「天文学の歴史的発見やドラマへの強い関心」
- 「生命存在の可能性など身近な視点での宇宙への興味」

題材選びには反応してくれている。**でも、出力品質で台無しになっていた**。これが Claude Code には拾いきれなかった部分です。

## 詰まったポイント

### 1. VOICEBOX の誤読は事後対応しかできない

VOICEBOX は合成音声なので、漢字を誤読しても**音としては流暢に出力**されます。特定の漢字を意図しない読みで発音する事象は、視聴者がコメントで指摘してくれて初めて気づくものでした。

Claude が組んでくれた仕組みには「読み辞書補正」というレイヤーがあって、誤読語を発見したら登録できる構造になっています。これ自体は良い設計でした。

問題は、**事後対応である以上、初出の動画は誤読のまま投稿されてしまう** ことです。日本語は読みのバリエーションが多く、誤読は無限に出てきます。

これは Claude Code の設計力の問題というより、**音声合成エンジンの根本的な限界** を Claude では解決できなかった、という話です。

### 2. AI が生成する台本は「過去形」になりがち

Claude API に台本を生成させていたのですが、学術的なトーンを意識した結果として、過去形・推量形が頻出していました。

> 「〜と考えられていた」
> 「〜だった」
> 「〜と推測されていた」

視聴者からすると「で、現在の最新説は何？」と気になる。プロンプトで「現在形・最新の知見も含めて」と何度も指示しましたが、完全には抑えきれませんでした。

LLM は教師データの影響を強く受けるので、Wikipedia 由来の文章スタイルに引っ張られがちです。これも個別に Claude Code を使って改善できる範囲を超えていました。

### 3. 「ガード」と「品質」は別物

Claude が組んでくれた4層ガードは、**同じ題材の重複** はほぼ完璧に防いでくれました。同じ天体を2日連続で扱う、みたいな事故はゼロ。

でも、**読み間違い** や **過去形だけの語り** といった、品質方向の問題は防げません。これは別レイヤーの話だからです。

設計を頼む側として、「重複防止」とは別に「品質保証」のレイヤーを最初から要求するべきでした。後から気づくのではなく。

## Claude Code に任せて1ヶ月で学んだこと

### 1. 仕組みを組むこと自体は早い

Docker, n8n, Python のサーバー、API 連携、Windows タスクスケジューラ登録用のコマンド作成まで含めて、Claude Code に頼んでから完全自動投稿が動くまで、思ったより短期間でした。コードを書く工数はほぼゼロで、私は仕様を伝えていただけです（タスクスケジューラの最終登録は管理者権限で自分で実行）。

### 2. でも「動いている」と「価値がある」は別

毎日自動で動画が投稿されている状態は、見ていて気持ちが良い。タスクスケジューラ通り発火し、毎日決まった時間に YouTube に動画が上がる。

でも視聴者は **出力されたもの** だけを評価します。「自動で投稿されているからすごい」ではなく、**「内容として面白いか」「ちゃんと聞き取れるか」** です。

### 3. AI に任せる時の盲点は「品質判定の主体」

Claude Code は私の指示通りに作ります。私が品質基準を明確にしなければ、品質はランダム化されます。

「視聴者が読み間違いに気づいてストレスを感じない動画」という基準を、最初から仕様に入れるべきでした。後から「やっぱり品質も自動で上げて」と頼んでも、それは別のシステムを作る話になります。

### 4. AI で副業を作る時のレッスン

副業として AI 自動化を考えるとき、「動くものができた」が **ゴールではなくスタート地点** だと、今は思います。

仕組み構築は Claude Code で済む。本当の差別化は、その先の **「視聴者・読者・ユーザーから見たときの不快感ライン」** を、人間が引いて、Claude にそこを守るように指示できるかどうか、です。

## このシステムから派生して書ける記事

このパイプラインを作ったときに副産物として得た「ニッチで日本語情報が少ない知見」を、別記事として書く予定です。

- VOICEBOX × Docker の接続罠（`host.docker.internal`）
- n8n の `workflow_history` 問題と DB 直接更新
- DockerからWindowsのFFmpegを呼ぶブリッジサーバー設計
- 重複防止4層ガードの設計思想
- VOICEBOX 誤読対策の読み辞書管理
- Sonnet 創作タスクで多様性を確保する温度設定とランダム制約

これらは Claude Code が書いたコードを Claude が解説する形になりますが、**実装事例として残しておく価値はある** と思っています。
